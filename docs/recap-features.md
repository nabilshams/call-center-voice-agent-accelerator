# Recap — Feature Specification

> **Recap** is an ambient clinical documentation assistant for clinicians. It listens to the clinician–patient conversation, produces a structured clinical note, and lets the clinician ask follow-up questions about the encounter. Inspired by Microsoft Dragon Copilot, built on Azure AI.

**Status:** Draft v0.1
**Owner:** TBD
**Starting point:** [server/static/recap.html](../server/static/recap.html) (forked from `live-transcription.html`) + `/recap` route in [server/server.py](../server/server.py).

---

## 1. Vision & Positioning

| | |
|---|---|
| **Problem** | Clinicians spend 1–2 hours per day on documentation; cognitive load during the visit is high; notes are often written hours later from memory. |
| **Solution** | Capture the visit ambiently, generate a structured note in the clinician's preferred template, and surface it for review/edit before it lands in the EHR. |
| **Primary user** | Outpatient clinicians (PCPs, specialists). Secondary: inpatient rounding, telehealth visits. |
| **Success metric (north star)** | Minutes of documentation time saved per encounter (target: > 4 min). |
| **Differentiators vs. generic transcription** | Clinical ASR, specialty-aware templates, EHR write-back, grounded clinical Q&A, personalization. |

---

## 2. Feature Map (Capability Tiers)

Tiered roughly by build order. Each tier is independently shippable.

### Tier 0 — Today (inherited from the accelerator)
Already working in `/recap`:
- Browser-mic audio capture (16 kHz PCM over WebSocket).
- Real-time partial + final transcription via Azure Speech SDK.
- LLM-based diarization & speaker-role inference (`Patient`, `Doctor`, etc.) via `gpt-4o-mini`.
- Free-text post-call summary with `key_points` and `action_items`.
- Transcript saved as JSON to Azure Blob.

### Tier 1 — MVP "Ambient SOAP Note"
The smallest thing that could plausibly replace a clinician's scratch notes.

### Tier 2 — Clinician productivity
Personalization, templates, draft-then-edit workflow, post-visit Q&A on the encounter.

### Tier 3 — Healthcare-grade
EHR integration, compliance posture, grounded knowledge, multi-language.

### Tier 4 — Platform
Org-wide rollout, analytics, admin tooling, API.

## 3. Tier 1 — MVP "Ambient SOAP Note"

### F1.1 — Encounter capture
- Start / Pause / Stop controls (Pause must keep the note in progress).
- Visual recording indicator + elapsed time.
- Local audio level meter (privacy reassurance: clinician can see it's listening).
- Auto-stop after configurable silence (default 2 min) to avoid runaway recordings.
- Mobile-friendly responsive layout (clinicians use phones in exam rooms).

### F1.2 — Live transcript with role labels
- Existing partial/final stream.
- Role-labeled bubbles: **Clinician** vs. **Patient** vs. **Other** (replaces generic "Speaker 1/2").
- Inline edit of any final utterance (text or speaker label) before note generation.
- Toggle: hide/show transcript pane (some clinicians want only the note).

### F1.3 — Structured SOAP note generation
On Stop (or "Generate Note" button), produce a structured note with these sections:

| Section | Content |
|---|---|
| **Chief complaint** | One short sentence. |
| **Subjective (S)** | HPI in narrative form, ROS bullets if mentioned. |
| **Objective (O)** | Vitals, exam findings explicitly stated by clinician. |
| **Assessment (A)** | Differential / working diagnosis with reasoning. |
| **Plan (P)** | Medications, tests, referrals, follow-up, patient education. |

Non-goals for MVP: ICD-10/CPT coding, billing-ready output.

### F1.4 — Note editor
- Section-by-section editable rich-text view.
- "Regenerate this section" button per section (re-prompts the LLM with just that section's instructions).
- "Copy to clipboard" + "Copy as plain text" + "Download .txt / .md".
- Show source-grounding: clicking a sentence in the note highlights the matching turns in the transcript.

### F1.5 — Confidence & safety guardrails
- Every section header shows a confidence chip (low/med/high) based on LLM self-rating.
- Banner on top of the note: *"Draft generated by AI. Review before clinical use."*
- Hallucination guard: any medication name / dose mentioned in the note must appear verbatim in the transcript; otherwise flagged.

### F1.6 — Persistence
- Save encounter (transcript + note + metadata) to Blob with structured key:
  `recap/{clinician_id}/{yyyy}/{mm}/{dd}/{encounter_id}.json`
- Local "My recent encounters" list (last 10, paginated).
- Encounter detail view with both transcript + note.

### F1.7 — Auth (minimum bar to be plausible)
- Microsoft Entra ID sign-in (no anonymous access to `/recap`).
- Single role for MVP: `Clinician`.
- All blob writes use the signed-in user's identity (no shared keys).

## 4. Tier 2 — Clinician productivity

### F2.1 — Specialty templates
Out-of-the-box templates beyond SOAP:
- **H&P** (history & physical, inpatient).
- **Progress note** (SOAP-shaped but shorter).
- **Procedure note**.
- **Telehealth visit**.
- **Consult letter** (referral reply).
Specialty packs (cardiology, derm, peds, etc.) override prompts and section ordering.

### F2.2 — Personalization
- Clinician profile: specialty, preferred template, signature block, common abbreviations.
- "Style learning": store accepted vs. edited sections; build a per-clinician few-shot prompt over time.
- Macros / dot-phrases (`.normexam` → expanded normal exam text).

### F2.3 — Post-visit Q&A on the encounter
Chat panel grounded in the encounter:
- "What did the patient say about chest pain duration?"
- "Draft a referral letter to cardiology."
- "Summarize for the patient (lay language)."
- Answers cite back to transcript turns.

### F2.4 — Draft-then-edit workflow
- Notes stay in `Draft` until clinician clicks **Sign**.
- Signed notes are immutable (versioned).
- Edit history retained for audit.

### F2.5 — Patient instructions / after-visit summary
One-click generation of:
- Patient-facing AVS in plain language (Flesch-Kincaid grade 6 target).
- Optional translation to Spanish, Mandarin, etc.

### F2.6 — Better diarization
- Move from LLM-batch diarization to streaming speaker-change detection.
- Optional pre-visit "voice enrollment" so the clinician's voice is reliably identified.
- Handle 3+ speakers (family member in room).

## 5. Tier 3 — Healthcare-grade

### F3.1 — EHR integration
- **Read**: pull patient demographics, problem list, meds, recent labs to ground the note.
- **Write**: push the signed note + orders + referrals back via SMART-on-FHIR / vendor APIs (Epic, Oracle Health, athenahealth).
- Per-tenant connector configuration; least-privilege scopes.

### F3.2 — Grounded clinical knowledge
- RAG layer over a curated medical-content corpus (e.g., uploaded guidelines, formulary).
- Drug-interaction check on any medication mentioned in the Plan.
- Citation links in answers and notes.

### F3.3 — Multi-language
- Recognition: en-US, en-GB, es-US/MX, fr-FR, de-DE, pt-BR (priority list).
- Note generation always in clinician's chosen language regardless of conversation language.
- Patient-facing AVS in patient's preferred language.

### F3.4 — Compliance & data handling
- HIPAA-ready architecture (BAA-eligible Azure services only).
- Customer-managed keys (CMK) on all storage.
- Configurable data residency (region pinning).
- Audit log of every read/write of encounter data, surfaced to admins.
- "No-train" guarantee: encounter data never leaves the tenant boundary; never used to train models.
- PHI redaction option for any data exported outside the tenant (analytics, support).

### F3.5 — Reliability
- Resume on network drop (audio buffered locally up to 5 min).
- "Offline degraded mode": capture audio, transcribe + summarize when reconnected.
- SLA target: 99.9% capture success, < 30 s p95 to first partial.

## 6. Tier 4 — Platform

### F6.1 — Admin console
- Tenant onboarding, user provisioning (SCIM).
- Template library management at org/dept level.
- Usage dashboards (encounters/clinician/day, time saved, edit rates per section).
- Cost dashboards (Azure consumption per encounter).

### F6.2 — Analytics for clinicians
- Personal dashboard: encounters this week, average note-edit time, top edited sections (signal for prompt tuning).

### F6.3 — Public API
- REST + webhooks for: start encounter, retrieve note, sign note, export.
- Lets partners embed Recap in their own scheduling/EHR tools.

### F6.4 — Marketplace integrations
- Microsoft Teams app (start a Recap from a Teams call).
- Outlook add-in (auto-create encounter from a calendar visit).
- Zapier / Power Automate connector.

## 8. Architecture deltas from today's accelerator

| Component | Today | Recap MVP |
|---|---|---|
| Auth | None on `/live-transcription` | Entra ID required on `/recap`; user identity propagated to backend |
| Storage layout | `transcripts/{persona}/...` | `recap/{clinician_id}/{yyyy}/{mm}/{dd}/{encounter_id}.json` |
| LLM call | Single summary prompt | Multi-section prompt chain (one per SOAP section) + guardrail validator |
| Persona inference | Generic personas (Customer/Agent/Patient/Doctor) | Fixed Clinician/Patient/Other; bias prompt accordingly |
| Diarization | LLM batch every 5 s | Same for MVP; upgrade in Tier 2 |
| Persistence model | Single JSON blob | Add `encounter` table (Cosmos DB) for listing/search; blob holds payload |
| Frontend | Single static page | React/Next.js app once we add encounter list, signing flow, editor |
| EHR | None | SMART-on-FHIR client (Tier 3) |

## 9. Risks & open questions

- **Regulatory classification** — Does Recap qualify as a medical device in any target market? (Likely no for documentation-only; revisit if we add diagnostic suggestions.)
- **Liability for AI-generated notes** — Need clear "draft, requires review" UX and audit trail.
- **EHR integration cost** — Each EHR connector is months of effort; pick one launch partner.
- **Specialty coverage** — Start with primary care; specialty packs need clinical SMEs.
- **Pricing model** — Per-clinician/month vs. per-encounter; depends on Azure cost per encounter (mostly OpenAI tokens + Speech minutes).
- **Hallucination risk in clinical content** — Guardrails in F1.5 are a start; need ongoing eval pipeline.

---

## 10. Out of scope (explicitly)

- Real-time clinical decision support / diagnostic suggestions.
- Billing & coding automation (ICD-10, CPT) — can come later, but big regulatory surface.
- Full EHR replacement.
- Patient-facing scheduling / portal features.
- Voice-controlled commands during the visit ("Recap, order CBC") — interesting but adds complexity and risk.
